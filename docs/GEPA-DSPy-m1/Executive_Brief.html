02 · The Mission

Why this work, and why now.

Four Blocks ships a product whose value lives almost entirely in language — the system prompts, the skill files, the way it speaks back. Until this sprint, every improvement to that language was an act of artisanal craft: someone read the output, formed a hypothesis, edited a prompt, and shipped it on intuition. That works at small scale. It does not scale.

The bet we made is simple: borrow the best academic work in the field — DSPy from Stanford and GEPA (Genetic-Pareto Prompt Evolution, ICLR 2026 Oral) — wire it to our own evaluation harness, and let the system propose its own improvements while humans stay in the review seat. No GPU training. No fine-tuning. Just continuous, traceable, reversible refinement of the language that drives the product.

Operating Principle

Every evolved variant must beat baseline on a held-out set the optimizer has never seen, pass our size and semantic-drift guardrails, and sit in front of a human before it ships. Optimization speed is good; auditable optimization is the actual product.

03 · The Three Pillars We Built

An evolutionary system needs three things to work.

A framework to evolve. A trustworthy dataset to grade against. A grading function that actually correlates with what we care about. We invested in all three.

⚙

The Optimizer

We adopted NousResearch's hermes-agent-self-evolution — DSPy + GEPA wired for skill, prompt, and tool optimization. We patched four files to bring it onto the latest DSPy 3.2 surface and to make body-only artifact validation work. The full test suite is green, and the CLI runs end-to-end against our infrastructure.

📚

The Corpus

We expanded and audited the canonical knowledge corpus — three new chapter sets joined the existing nine, bringing the system to 159 verbatim, source-attributed chunks. A nine-chunk fidelity sample passed end-to-end. Cross-references resolve cleanly, with zero pointers to excluded material.

◐

The Architecture

We formalized a hybrid two-layer architecture separating what the model retrieves from (verbatim source) from how it speaks (brand voice). This lets us train on unparaphrased ground truth without leaking attribution into user-facing copy — a clean division that didn't exist before.

⛨

The Guardrails

Every evolution run passes through size limits, growth caps, and structural validation before any human even sees the diff. We've codified the rules — ≤15KB skill bodies, ≤20% growth over baseline, mandatory human review — and wired them into the test harness so they can't quietly drift.

04 · The Hybrid Two-Layer Architecture

Two layers. One clean separation of concerns.

The single most consequential design decision of this sprint. It quietly resolves three problems that would have surfaced later: training fidelity, brand voice consistency, and attribution discipline.

Layer 1 — The Ground Truth

Internal · Retrieval Source

Verbatim, source-attributed chunks. This is what the system retrieves from and what the evaluator scores against. It is never paraphrased and never user-facing. Training pipelines always read from here.

Layer 2 — The Voice

External · Response Style

Distilled, attribution-clean, on-brand language. This is what end users hear back. It is formally tagged as not a training target — using it would create a paraphrase-of-paraphrase loop and silently drift the system off-source.

05 · The Engineering Story

Four targeted patches. The framework went from "demo" to "production-runnable."

The off-the-shelf optimizer didn't run cleanly against our setup. Rather than wait for an upstream fix, we patched it ourselves — surgically, with tests — and now have a working path that we can also offer back upstream.

evolution/core/
constraints.py

Body-only artifact validation

The structure validator was rejecting every evolved variant because GEPA produces a markdown body without frontmatter. We taught the validator to accept body fragments while still requiring full SKILL.md files to carry their headers.

evolution/skills/
evolve_skill.py

DSPy 3.2 GEPA upgrade

Migrated the optimizer call to the modern DSPy interface — added the five-argument metric bridge, swapped step counting for full-evaluation budgeting, wired in a reflection model, and used the keyword-style student signature. End-to-end runs now succeed.

tests/core/
test_constraints.py

Test alignment

The constraint test suite was inverted relative to our new behavior. Updated assertions to lock in the body-fragment rule. All 139 tests pass under the new contract.

pyproject.toml

Fallback path stabilized

Added Optuna to the dev dependency set so the secondary optimizer (MIPROv2) is always available as a fallback. Future contributors get a clean install on the first try.

A note on this number

This run used the framework's default keyword-overlap fitness function — a fast, cheap proxy. For the production block work, we'll graduate to an LLM-as-judge metric scored against Layer 1 source chunks, which should produce more meaningful (and more conservative) lift numbers. 07 · Guardrails — The Discipline Around the Speed

Speed without discipline is a regression machine.

Three guardrails are in force today. They're cheap, they're testable, and they make sure an evolution run can never quietly ship something that drifts off-mission.

✓

Size cap · ≤ 15KB

Skill bodies cannot balloon under optimizer pressure. The 15KB ceiling matches the upstream contract and is enforced before any variant is considered a candidate.

✓

Growth limit · ≤ 20%

No variant may grow more than 20% over its baseline. This is a soft semantic-drift signal: if the optimizer is doubling the prompt, it's almost certainly fitting noise.

✓

Human-in-the-loop

Every evolved variant produces a reviewable diff plus a holdout report. Nothing auto-commits to production prompts. The optimizer proposes; a person disposes.

09 · Why This Matters

The shift is from craft to system.

A month ago, improving Four Blocks meant a thoughtful person sitting down with a prompt and iterating by feel. That work is good — but it doesn't compound, doesn't audit cleanly, and doesn't survive personnel changes.

What we built this sprint is a system. The corpus is auditable. The optimizer is reproducible. The guardrails are testable. The architecture separates what we know from how we say it. And the scientific machinery — held-out sets, metric-driven selection, human review gates — is borrowed directly from the best evaluation practice in the field.

The product hasn't changed yet. The trajectory has. Every future improvement to Four Blocks will now leave a measured paper trail, with an unbroken line back to the source material the framework was built on. That's the bet. The first +22% says it's working.
